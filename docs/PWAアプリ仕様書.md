# シフトリマインドBot PWAアプリ 仕様書

## 1. プロジェクト概要

| 項目 | 内容 |
|------|------|
| プロジェクト名 | BizCAFE シフトリマインド PWAアプリ |
| 目的 | シフト管理・リマインド機能をスマホ/iPadからも操作可能にする |
| アプリ形式 | **PWA（Progressive Web App）** — ブラウザベース、ホーム画面追加でアプリ風に動作 |
| 利用者 | 店長（管理者）+ スタッフ全員 |
| 技術基盤 | **Google Apps Script（API） + HTML/CSS/JS（フロントエンド）** |
| コスト | **月額 0円**（GAS + GitHub Pages or GAS HTML Service） |

#### アップロードフロー
1. ユーザーがPDFファイルを選択（`<input type="file" accept=".pdf">`）
2. `FileReader` でBase64に変換
3. GAS API (`action=uploadPdf`) に送信
4. GAS側でPDF → OCR → テキスト解析 → 全日分のシフトデータを抽出
5. 解析結果をプレビュー表示
6. 「シフトを保存する」で Google Sheets に保存
7. 保存済みPDFを Google Drive の処理済みフォルダへ移動

### 4.8 LINE WORKS トークによるシフト変更申請

スタッフがLINE WORKSのトークルームで特定のフォーマットで発言することで、シフト変更を自動反映する。

**フォーマット:**
```
【シフト変更】
日付: 2026/02/14
名前: 杉本尚哉
変更: 15:30〜18:00
```
または
```
【シフト変更】
日付: 2026/02/14
名前: 杉本尚哉 → 武田祥哉
変更: 15:30〜18:00
```

**ルール:**
- **締め切り**: 対象シフト**前日の11:30**まで。
  - 11:30以降の投稿は「受付時間外」としてエラー返信。
- **反映処理**: Webhookで受信 → 解析 → `シフト`シートを更新。
- **完了通知**: 更新成功時、Botが「変更を受け付けました」と返信。

### 4.9 シフト不足アラート

**仕様:**
- **対象**: 5日後のシフト
- **条件**: 営業日かつ、営業時間内でスタッフが1人もいない時間帯がある場合。
- **通知**: 毎日10:00（定期実行）。
- **内容**: `@all` メンションで不足日時を通知。

**メッセージ例:**
```
🚨 【シフト不足のお知らせ】
2月19日(水) のシフトが不足しています！

不足時間帯: 14:00〜15:30
@all ご協力をお願いします！
```

## 5. データ設計（Google Sheets）

### 5.1 シート一覧

既存のGASバックエンド（自動リマインド）と PWAアプリで**同一のSpreadsheetを共有**する。

| シート名 | 用途 | 読み書き |
|----------|------|----------|
| `シフト` | シフトデータ（PDF解析結果を保存） | 読み書き |
| `スタッフ` | スタッフ名 ↔ LINE WORKSアカウントマッピング | 読み書き |
| `送信ログ` | リマインド送信履歴 | 書き込み・読み取り |
| `営業時間` | 営業時間の定義（履歴管理） | 読み取り |← 追加
| `設定` | アプリ設定値（PIN等） | 読み取り |

### 5.2 シフトシート

| 列 | カラム名 | 型 | 例 |
|----|---------|-----|-----|
| A | 日付 | Date | 2026/02/14 |
| B | 曜日 | String | 土 |
| C | スタッフ名 | String | 杉本尚哉 |
| D | 開始時刻 | String | 15:30 |
| E | 終了時刻 | String | 17:30 |
| F | 業務内容 | String | 在報・棚卸 / 清掃 |
| G | 登録日時 | DateTime | 2026/02/10 14:00:00 |
| H | 登録元 | String | pdf_upload / manual |

### 5.3 スタッフシート

| 列 | カラム名 | 型 | 例 |
|----|---------|-----|-----|
| A | スタッフ名 | String | 杉本尚哉 |
| B | LINE WORKSアカウントID | String | sugimoto@bizcafe.com |
| C | 有効 | Boolean | TRUE |
| D | 登録日 | Date | 2026/01/15 |

### 5.4 送信ログシート

| 列 | カラム名 | 型 | 例 |
|----|---------|-----|-----|
| A | 送信日時 | DateTime | 2026/02/13 12:00:05 |
| B | 対象日 | Date | 2026/02/14 |
| C | スタッフ数 | Number | 2 |
| D | 送信方法 | String | auto / manual |
| E | 結果 | String | success / error |
| E | 結果 | String | success / error |
| F | 詳細 | String | 正常送信 / エラーメッセージ |

### 5.5 営業時間シート（新規）

不定期な営業時間変更に対応するため、開始日ベースで管理する。

| 列 | カラム名 | 型 | 例 | 説明 |
|----|---------|-----|-----|-----|
| A | 開始日 | String | 20260225 | この日付以降（次の設定日まで）適用 |
| B | 開始時刻 | String | 11:00 | 営業開始 |
| C | 終了時刻 | String | 19:00 | 営業終了 |
| D | 備考 | String | 春休み期間 | メモ |

**判定ロジック:**
対象日のシフトを確認する際、対象日 <= 開始日 となる設定の中で、最も開始日が新しい（大きい）レコードを採用する。

例:
- 行1: `20260201`, `10:00`, `20:00`
- 行2: `20260225`, `11:00`, `19:00`

2/14の営業時間 → 行1 (20260201) を適用。
2/26の営業時間 → 行2 (20260225) を適用。

### 5.5 設定シート

| 列 | カラム名 | 型 | 例 |
|----|---------|-----|-----|
| A | キー | String | manager_pin |
| B | 値 | String | 1234 |

設定値一覧:

| キー | 説明 | 初期値 |
|------|------|--------|

## 6. API設計（GAS Web App）

### 6.1 共通仕様

| 項目 | 内容 |
|------|------|
| ベースURL | GAS Web AppのデプロイURL |
| リクエスト | `doGet(e)` — 読み取り系 / `doPost(e)` — 書き込み系 |
| レスポンス | JSON (`ContentService.MimeType.JSON`) |
| 認証 | リクエストパラメータに `token` を付与 |
| CORS | GAS Web Appは同一ドメインではないため `doPost` 利用時は注意 |

### 6.2 エンドポイント一覧

#### GET — `doGet(e)`

| action | 権限 | 説明 | パラメータ |
|--------|------|------|-----------|
| `auth` | なし | PIN認証 | `pin` |
| `getShifts` | 全員 | シフトデータ取得 | `token`, `from`(日付), `to`(日付) |
| `getStaff` | 店長 | スタッフ一覧 | `token` |
| `getLogs` | 店長 | 送信ログ取得 | `token`, `limit`(件数) |
| `getPreview` | 店長 | 送信プレビュー | `token`, `date` |

#### POST — `doPost(e)`

| action | 権限 | 説明 | ボディ |
|--------|------|------|--------|
| `uploadPdf` | 店長 | PDFアップロード・解析 | `token`, `fileBase64`, `fileName` |
| `saveParsedShifts` | 店長 | 解析済みシフトを保存 | `token`, `shifts[]` |
| `updateStaff` | 店長 | スタッフ追加/編集 | `token`, `name`, `accountId` |
| `deleteStaff` | 店長 | スタッフ削除 | `token`, `name` |
| `sendReminder` | 店長 | 手動リマインド送信 | `token`, `date` |

## 8. 既存バックエンドとの統合

### 8.1 変更点

既存の自動リマインドバックエンド（`src/`）に以下の変更を加える：

| 変更箇所 | 内容 |
|----------|------|
| データソース変更 | `config.gs` のハードコード → **Google Sheetsから読み取り** |
| スタッフマッピング | `STAFF_MAPPING` 定数 → **「スタッフ」シートから取得** |
| シフトデータ取得 | PDF直接解析 → **「シフト」シートから取得**（PDFはアプリ経由でアップロード済み） |
| 送信ログ記録 | `Logger.log` のみ → **「送信ログ」シートにも記録** |

### 8.2 自動リマインドフロー（変更後）

```
[毎日12:00 GASタイムトリガー]
        |
        v
[「シフト」シートから翌日分を取得]
        |
        v
[「スタッフ」シートからマッピング取得]
        |
        v
[LINE WORKS Bot API でメンション付き送信]
        |
        v
[「送信ログ」シートに記録]
```

- PDFの直接解析は廃止 → **アプリからのアップロード時に1回だけ解析し、Sheetsに保存**
- 自動リマインドはSheetsのデータを読むだけのシンプルな処理に

### 8.3 PDFアップロードのワークフロー（新規）

```
[店長がアプリからPDFをアップロード]
        |
        v
[GAS API: Base64 → Blob → Drive保存]
        |
        v
[PDF → Googleドキュメント変換（OCR）]
        |
        v
[テキスト解析 → 全日分のシフトデータ抽出]
        |
        v
[解析結果をアプリに返却（プレビュー表示）]
        |
        v
[店長が確認 → 「保存」ボタン]
        |
        v
[「シフト」シートに全日分を一括保存]
        |
        v
[PDFをDrive処理済みフォルダに移動]
```

## 13. 注意事項

- GAS Web Appの `doPost` は初回アクセス時にcold startで数秒遅延する場合がある
- PDFアップロードはGASの実行時間制限（6分）に注意 → 大きなPDFは分割処理を検討
- Google Sheetsの同時書き込みは競合の可能性あり → ロック処理（`LockService`）を使用
- Service Workerのキャッシュ更新戦略は「Network First, Cache Fallback」を採用
- スタッフマッピングをSheetsに移行後、`config.gs` の `STAFF_MAPPING` 定数は廃止する
