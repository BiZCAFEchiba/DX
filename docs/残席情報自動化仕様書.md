# 残席情報自動更新システム 仕様書

## 1. 概要

| 項目 | 内容 |
|------|------|
| 機能名 | 残席情報自動更新 |
| 目的 | スタッフによる手動更新を廃止し、QRスキャンログから残席数をリアルタイム自動算出する |
| 対象 | BizCAFE 千葉大学店（全62席） |
| 利用者 | スタッフ（QRスキャン） / 顧客（残席表示閲覧） |
| 技術基盤 | GAS Web App（API） + Cloudflare Workers PWA + Googleスプレッドシート |

---

## 2. 要件定義

### 2.1 ビジネスルール

| ルール | 内容 |
|--------|------|
| 総席数 | 62席 |
| クールダウン | shirucafeアプリ側で **90分間** の再提示制限を管理（BizCAFE側では追跡不要） |
| 在席判定 | 直近90分以内にスキャンされた回数 ＝ 在席中の顧客数 |
| 残席算出 | **残席数 ＝ 62 − 直近90分以内のスキャン件数** |
| 手動上書き | スタッフが任意の残席数を手動設定可能。次のスキャン発生時に自動計算に戻る |

### 2.2 顧客側の仕組み（shirucafeアプリ） ※アプリ画面・QR内容確認済み

- 顧客は注文時に **shirucafeアプリ上のQRコード** をスタッフに提示する
- QRコードは以下のURL形式のワンタイムリンク：
  `https://mypage.shirucafe.com/store/staff?access_token=xxx&in_out=eatin&...`
- `in_out=eatin`（イートイン）のみ座席占有としてカウント
- `in_out=takeout`（テイクアウト）はカウントしない
- 90分のクールダウンは **shirucafeアプリ側** で制御済み → BizCAFE側での顧客ID追跡は不要
- `access_token` がワンタイムトークン → 同一QRの二重スキャン防止に使用

### 2.3 スタッフ側の操作フロー

```
顧客がQRコードを提示
        |
        v
スタッフがスタッフPWAのQRスキャン画面でカメラを起動
        |
        v
QRコードを読み込む（customer_id を取得）
        |
        v
API へスキャンを送信（POST /api/qr_scan）
        |
        v
GASがスキャンログを記録 → 残席数を自動再計算
        |
        v
顧客アプリの残席表示が更新される
```

## 4. データ設計

### 4.1 QRスキャンログシート（新規作成）

シート名: `QRスキャンログ`

| 列 | 列名 | 型 | 説明 |
|----|------|----|------|
| A | スキャン日時 | DateTime | スキャンされた日時（JST） |
| B | access_token | String | QRURLから抽出したワンタイムトークン（重複チェック用）|
| C | in_out | String | `eatin` or `takeout` |
| D | スタッフ名 | String | スキャンを行ったスタッフ名 |
| E | 在席期限 | DateTime | スキャン日時 + 90分 |
| F | drink_id | String | ドリンクID（任意記録） |

**ポイント:**
- B列 `access_token` で二重スキャン防止
- C列 `in_out = eatin` の行のみ在席カウント対象
- A列（スキャン日時）で直近90分を絞り込み、C列がeatiの件数 = 在席数

### 4.2 残席設定シート

既存の `設定` シートに以下の行を追加:

| キー | 値 | 説明 |
|------|-----|------|
| `残席手動上書き` | 数値 or 空白 | 空白なら自動計算、数値なら手動上書き |
| `残席上書き期限` | DateTime | 上書き有効期限（省略時は次スキャンまで有効） |

## 5. API設計

### 5.1 QRスキャン記録

```
POST /gas_api
Body: {
  "action": "qrScan",
  "qrUrl": "https://mypage.shirucafe.com/store/staff?access_token=xxx&in_out=eatin&...",
  "staffName": "田中勇平"
}

Response（イートインの場合）: {
  "ok": true,
  "remainingSeats": 45,
  "isNewScan": true,        // false = 同一QRの二重スキャン
  "isEatIn": true,          // false = テイクアウト（カウントなし）
  "occupiedSeats": 17
}

Response（テイクアウトの場合）: {
  "ok": true,
  "remainingSeats": 45,     // 変化なし
  "isNewScan": true,
  "isEatIn": false,
  "occupiedSeats": 17
}
```

**GAS側処理:**
1. `qrUrl` から `access_token`・`in_out` を抽出
2. `access_token` がスキャンログに既に存在するか確認
   - 既存 → `isNewScan: false` を返す（ログに記録しない）
3. `in_out !== 'eatin'` の場合 → `isEatIn: false` を返す（ログに記録しない）
4. 新規イートイン → スキャンログに追記（日時・access_token・in_out・スタッフ名・在席期限）
5. 直近90分以内のeatin件数を集計して残席数を計算

### 5.2 残席数取得

```
GET /gas_api?action=getSeats

Response: {
  "ok": true,
  "remainingSeats": 45,
  "totalSeats": 62,
  "occupiedSeats": 17,
  "isManualOverride": false,         // true = 手動上書き中
  "updatedAt": "2026-04-16T14:55:00+09:00"
}
```

**GAS側処理:**
1. 手動上書きが有効 → 上書き値を返す
2. 自動モード → 直近90分以内のスキャンログからユニーク顧客数を集計
3. 残席 = 62 − ユニーク顧客数

### 5.3 手動上書き

```
POST /gas_api
Body: {
  "action": "seatsOverride",
  "seats": 30,          // 上書きする残席数（0〜62）
  "staffName": "星野竜大"
}

Response: { "ok": true, "remainingSeats": 30, "isManualOverride": true }
```

### 5.4 手動上書き解除

```
POST /gas_api
Body: { "action": "clearSeatsOverride" }

Response: { "ok": true, "remainingSeats": 45, "isManualOverride": false }
```

## 6. 残席計算ロジック

```javascript
function calcRemainingSeats() {
  const TOTAL_SEATS = 62;

  // 手動上書きチェック
  const override = getSeatsOverride_();
  if (override !== null) return { seats: override, isManual: true };

  const now = new Date();
  const cutoff = new Date(now.getTime() - 90 * 60 * 1000); // 90分前

  const sheet = SpreadsheetApp.openById(SPREADSHEET_ID)
                               .getSheetByName('QRスキャンログ');
  if (!sheet || sheet.getLastRow() < 2) return { seats: TOTAL_SEATS, isManual: false };

  const data = sheet.getDataRange().getValues();
  let occupied = 0;

  for (let i = 1; i < data.length; i++) {
    const scanTime = data[i][0]; // A列: スキャン日時
    const inOut    = data[i][2]; // C列: in_out
    if (!(scanTime instanceof Date)) continue;
    if (scanTime < cutoff) continue;            // 90分より前はスキップ
    if (String(inOut).trim() !== 'eatin') continue; // テイクアウトはカウントしない
    occupied++;
  }

  return { seats: Math.max(0, TOTAL_SEATS - occupied), occupied, isManual: false };
}

function recordQrScan_(qrUrl, staffName) {
  const url    = new URL(qrUrl);
  const token  = url.searchParams.get('access_token');
  const inOut  = url.searchParams.get('in_out') || '';
  const drink  = url.searchParams.get('drink')  || '';

  if (!token) return { ok: false, error: 'no_token' };

  const sheet = SpreadsheetApp.openById(SPREADSHEET_ID)
                               .getSheetByName('QRスキャンログ');

  // 二重スキャンチェック
  const data = sheet.getDataRange().getValues();
  for (let i = 1; i < data.length; i++) {
    if (String(data[i][1]).trim() === token) {
      return { ok: true, isNewScan: false, isEatIn: inOut === 'eatin' };
    }
  }

  // eatin のみ記録（テイクアウトはログに残さない）
  if (inOut === 'eatin') {
    const now      = new Date();
    const deadline = new Date(now.getTime() + 90 * 60 * 1000);
    sheet.appendRow([now, token, inOut, staffName, deadline, drink]);
  }

  return { ok: true, isNewScan: true, isEatIn: inOut === 'eatin' };
}
```

## 7. フロントエンド設計

### 7.1 スタッフPWA — QRスキャン画面

**新規ビュー: `views/qrScan.js`**

| 要素 | 内容 |
|------|------|
| カメラ起動ボタン | `getUserMedia` でカメラを起動 |
| QR読み取り | `jsQR` ライブラリ（軽量・無料）でQRコードをデコード |
| スタッフ名選択 | スタッフドロップダウン（既存の `allStaff` を流用） |
| 結果表示 | スキャン成功・二重スキャン・テイクアウト・エラーをトースト表示 |

**読み取りフロー:**
```
カメラ映像をcanvasに描画（30fps）
  → jsQR でフレームごとにQRコードを検出
  → QRコード検出 → URL文字列を取得
  → shirucafe の URL か検証（mypage.shirucafe.com）
  → API.qrScan({ qrUrl, staffName }) を呼び出す
  → 結果に応じてトースト表示:
     - isNewScan: true  + isEatIn: true  → ✅ スキャン完了（残席: XX席）
     - isNewScan: true  + isEatIn: false → 🥤 テイクアウト（残席変化なし）
     - isNewScan: false                  → ⚠️ 既にスキャン済みのQRコードです
     - エラー                            → ❌ 無効なQRコードです
```

### 7.2 スタッフPWA — 残席管理画面

**既存ダッシュボードまたは新規ビューに追加**

## 9. 手動上書きの動作仕様

| 状態 | 動作 |
|------|------|
| 上書き値が設定されている | `getSeats` は上書き値を返す |
| スタッフが上書き解除 | 即座に自動計算に戻る |
| QRスキャンが発生したとき | 自動計算値も常に裏で更新（表示はまだ上書き値） |
| 上書き期限が設定されていた場合 | 期限を過ぎると自動的に自動計算モードに戻る |

**用途例:**
- 臨時でイベントがあり席数が減る → 手動で低い値に設定
- システムの誤カウントが発生した → 手動で実際の残席数を入力して調整
- 営業終了 → 上書きで62（満席解除）にして翌日に備える

## 10. エラーハンドリング

| ケース | 対応 |
|--------|------|
| QRコードの読み取り失敗 | トーストで「読み取り失敗。再度試してください」 |
| customer_idの形式が不正 | API側でバリデーション → `{ ok: false, error: "invalid_qr" }` |
| スキャンログシートが存在しない | GAS側で自動作成 |
| 残席数がマイナスになる場合 | `Math.max(0, 62 - occupied)` で0以下にならないよう制御 |
| 顧客アプリのポーリング失敗 | 前回取得値を表示し続け、次回ポーリングで再試行 |

## 12. 技術的補足

### 12.1 jsQR について

- ライブラリURL: `https://cdn.jsdelivr.net/npm/jsqr/dist/jsQR.js`
- ブラウザのカメラAPIでキャプチャした映像フレームをデコード
- 追加費用なし・サーバー不要

### 12.2 ポーリング間隔の考え方

| 対象 | ポーリング間隔 | 理由 |
|------|--------------|------|
| 顧客アプリ | 3分 | 残席は急変しにくく、GAS負荷を抑える |
| スタッフ管理画面 | 30秒 | スタッフはリアルタイム性を重視 |

### 12.3 パフォーマンス

- QRスキャンログが蓄積した場合の対策:
  - 90日以上前の古いログを定期的に削除（月次GASトリガーで自動クリーンアップ）
  - または当日分のみ別シートに保持するアーキテクチャも可

### 12.4 shirucafe QRコードの形式（実機確認済み）

QRコードは以下のURL形式：
```
https://mypage.shirucafe.com/store/staff
  ?id=517121
  &drink=288
  &cup=1
  &in_out=eatin        ← eatin / takeout
  &access_token=c788b1db5b9e583da1c5ce06d4a2326d  ← ワンタイムトークン
  &ad=568
```

| パラメータ | 用途 |
|----------|------|
| `access_token` | **ワンタイムトークン**（一意識別子）。BizCAFEシステムでの重複スキャン防止に使用 |
| `in_out` | `eatin` のみ座席カウント対象。`takeout` は在席カウントしない |
| `id` | 注文ID（shirucafe側の識別子） |
| `drink` / `cup` | ドリンク情報（必要に応じて記録） |

**BizCAFEシステムでの抽出方法（JavaScript）:**
```javascript
const url = new URL(qrText); // QRから読み取ったURL文字列
const token   = url.searchParams.get('access_token'); // ワンタイムトークン
const inOut   = url.searchParams.get('in_out');       // "eatin" or "takeout"
const drinkId = url.searchParams.get('drink');
```
